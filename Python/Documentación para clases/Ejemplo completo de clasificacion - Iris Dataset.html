1. Carga de datos¶

El dataset de Iris está disponible directamente dentro de sklearn; así que no tendremos que descargarlo ni leerlo de un .csv ni nada por el estilo. Está dentro de sklearn.datasets:

In [4]:
from sklearn.datasets import load_iris

El dataset viene con una descripción, que imprimimos a continuación. Se trata de un dataset donde el objetivo es clasificar correctamente tres especies de plantas: Iris Setosa, Iris Versicolor e Iris Virginica. Es, por tanto, un problema de clasificación multiclase (a clasificar entre tres posibles clases, y no dos como en los ejemplos de la documentación).

El dataset cuenta con 150 observaciones, donde cada una son mediciones que se hicieron de un ejemplar. Dichas mediciones (features) de las que disponemos para predecir la especie son el tamaño del sépalo y pétalo de las plantas.

El objetivo es entrenar un clasificador, de forma que en el futuro si tenemos una de las plantas delante y una regla para medir el tamaño del sépalo y pétalo, seamos capaces de identificar (predecir) cuál de las especies es (si es Setosa, Versicolor o Virginica).

Pues nada. teniendo el problema claro, vamos a cargar el dataset en un DataFrame de pandas.

La función load_iris() devuelve un diccionario con varias claves:

  • DESCR, que lo hemos usado arriba, es un string con la descripción del dataset
  • data contiene las features como un array (matriz) de Numpy
  • feature_names es una lista de Python con los nombres de cada feature, ordenados en el mismo orden que las columnas de la matriz data
  • target contiene la variable a predecir (las 3 especies) como especie 0, especie 1 o especie 2
  • target_names contiene la correspondencia de 0-Setosa, 1-Versicolor, 2-Virginica

Así que nada: vamos muy rápidamente a usar todo esto para construir nuestro DataFrame:

In [6]:
# Cargamos las features en un DataFrame:
iris_dataset = pd.DataFrame(load_iris()["data"],
                            columns=load_iris()["feature_names"])

# Y añadimos la columna de especies:
iris_dataset["label"] = load_iris()["target_names"][load_iris()["target"]]
In [7]:
# Convertimos el DataFrame a jerárquico, para ser más organizados:
iris_dataset.columns = pd.MultiIndex.from_tuples(list(zip(["features"]*4, load_iris()["feature_names"]))
                                                 + [("label","species")])

3. Separación en train y test¶

Nos encontramos en un problema supervisado de clasificación, donde el objetivo es crear un clasificador capaz de discriminar/predecir entre tres clases de especies: setosa, versicolor y virginica; estamos por tanto ante un problema de clasificación multiclase (o sea: no de clasificación binaria).

Puesto que sklearn (y en general en machine learning) las máquinas solo entienden de números, vamos a convertir la columna label (que ahora mismo son strings con los nombres de cada especie) a números correspondientes (por ejemplo: 0 para una especie, 1 para otra y 2 para la otra). Cierto es que en origen ya venían como 0, 1 y 2; pero simplemente por fines demostrativos vamos a pretender que las clases a predecir venían como strings, y queremos cambiarlas a números.

Podríamos hacer esto directamente con pandas y el método .apply() que tienen las Series (columnas) de pandas; pero sklearn también tiene una funcionalidad más automática que lo hace para nosotros de forma muy sencilla: sklearn.preprocessing.LabelEncoder:

In [13]:
from sklearn.preprocessing import LabelEncoder

# Es super fácil de usar. No toma argumentos.
# Simplemente creamos una instancia:
label_encoder = LabelEncoder()

# Su método .fit() le permite identificar
# todas las distintas clases que hay:
label_encoder.fit(iris_dataset["label","species"])

# Y para transformar la columna en nuestro DataFrame,
# utilizamos el método .transform():
iris_dataset["label","species"] = label_encoder.transform(iris_dataset["label","species"])

4. Discusión sobre posibles pre-procesados del conjunto de train¶

El conjunto de test ya está separado, y no vamos a tocarlo ya hasta el final. Solo trabajamos con el conjunto de train (del cual iremos sacando folds de validación cruzada cuando hagamos Grid Search y toda la pesca...)

Pero antes de ponernos a hacer modelos, validaciones cruzadas y similares, deberíamos plantear, ahora sí, una serie de decisiones sobre nuestros datos:

  1. ¿Descartar variables directamente por alta correlación?
  2. ¿Quitar outliers?
  3. ¿Estandarizar las features?
  4. ¿Seleccionar features/hacer reducción de dimensionalidad?

Lo ideal sería probar todas estas combinaciones; especialmente si no conocemos bien los datos.

Eso sí: Si vamos a hacer validación (cross validation por ejemplo), así como Grid Search, es recomendable aplicar estos pre-procesados exclusivamente sobre los conjuntos de train, y no sobre los de validación. ¿Por qué? Por la misma razón que no lo debemos hacer sobre el conjunto de test. Los modelos, así como todos los posibles pre-procesados que les acompañan, solo deben ser entrenados sobre el conjunto de train, con el que el modelo entrena. Si no lo hacemos así, filtramos información a los modelos más allá de la de los datos con los que entrenan (y eso no deja de ser otra forma de sobreajuste).

Esto quiere decir que las 4 técnicas anteriores deberían ser empotradas en el propio proceso del Grid Search. Y sí: la forma más automática y sencilla es mediante el uso de Pipelines. Hay ocasiones en las que la gente no las empotra; y si bien no está del todo bien, el esfuerzo de empotrar dichas técnicas en el proceso de Grid Search es algo manual y rollo, y la gente prefiere "pecar" y no ser completamente purista con su metodología.

En este caso, vamos a considerar exclusivamente los temas de estandarizar y seleccionar features. Outliers no hay (deberíamos de comprobarlo ahora exclusivamente con el conjunto de train), reducción de dimensionalidad con menos de 100 features es estúpido, y hemos quedado en que no vamos a eliminar variables por alta correlación.

4.1. Estandarizar features¶

Esta decisión es sencilla: en todos los modelos de sklearn debemos estandarizar las features, excepto en los árboles (y sus ensembles, como Random Forest y GBTs) y en Naïve Bayes. Así que cuando probemos distintos modelos, dependiendo de cuál sea, deberemos añadir antes el StandardScaler o no.

4.2. Seleccionar features¶

En la teoría nos centramos mucho en cómo funcionan cada una de las técnicas de selección de variables; pero no tanto en el rationale detrás de cuándo y dónde hacerlo.

Empezemos por lo sencillo: básicamente, existen ciertos algoritmos (de clasificación y de regresión principalmente) en los que la selección de features son parte del propio entrenamiento del modelo. ¿En cuáles? Pues esencialmente en los árboles y sus ensembles. Al fin y al cabo, los árboles van haciendo splits y separando nuestro dataset feature por feature, en función de qué feature es la más discriminante en cada nivel de profundidad del árbol. Si un árbol (o un conjunto de ellos) deciden no hacer split por una determinada feature nunca, es una señal de que dicha feature probablemente sea irrelevante para nuestro problema. Así que en general, si el modelo que vamos a usar es uno basado en árboles, por lo general podemos ahorrarnos esta etapa del pre-procesado; a menos que tengamos miles de features; en cuyo caso a los árboles les cuesta mucho seleccionar y entrenar. En dichos casos, podemos acudir a otra técnica de selección previa al árbol, o a reducción de dimensionalidad. Pero hasta unos cuantos cientos de features, los árboles son capaces de seleccionar bien por motu propio.

Para el resto de modelos, sí suele ser conveniente seleccionar features con alguna (o varias) de las técnicas explicadas en la teoría. Quizás para las redes neuronales también nos lo podamos ahorrar; pero depende muchísimo de la naturaleza de nuestros datos. Si éstos son armoniosos y similares a las percepciones humanas (clasificación de imágenes, texto, vídeo...) no suele ser necesario. De todos modos, ¿cómo saber si debemos hacerlo o no? La solución es la de siempre: validación :)

Vale: y para el resto de modelos sí suele ser necesario. Pero...

  1. ¿Incluso aunque tengamos solo 4 features debemos seleccionar?
  2. ¿Y con cuántas nos quedamos?
  3. ¿Lo hacemos antes, o después de estandarizar las features?

La respuesta a la primera pregunta es básicamente no. Esencialmente todos los modelos tanto de clasificación como de regresión modernos pueden lidiar con unas cuantas decenas de features sin que resulte un problema. Si llegas a una cantidad considerable (por encima de 50), igual comienza a ser momento de valorar la selección de features.

La segunda pregunta es muy difícil de responder y depende muchísimo de los datos y el modelo que utilicemos. ¿Mi recomendación? Ante la duda, RFECV es lo que mejor funciona (y que el modelo seleccionador del RFECV sea el mismo que el clasificador/regresor que utilicemos luego), y no tenemos que tomar la decisión de con cuántas quedarnos: el loop interno de validación cruzada que tiene dicha técnica lo decide por nosotros. Para otras técnicas como RFE o SelectKBest, lo mejor es añadir dicho elemento a nuestra Pipeline, que a su vez entre en el GridSearchCV, y probar a quedarse con más/menos features.

Para la tercera pregunta solo tenemos que utilizar el sentido común. ¿Nuestra técnica de selección de features contiene/incluye el entrenamiento de algún modelo? Y... ¿Dicho modelo es sensible a la estandarización de features? Pues ya está respondido :) Y no: nunca estandarices dos veces (antes y después de la selección).

Dicho todo esto, y aunque no sea necesario en absoluto (puesto que tenemos solo 4 features), vamos a probar a incluir selección de features en algunas de las Pipelines que haremos.

Y poco más. Vamos al lío, y vamos a preparar toda la maquinaria para probar muchísimas combinaciones de pre-procesados y modelos:

8. Mediciones sobre el conjunto de test¶

Con nuestro mejor modelo ya seleccionado y entrenado con todo el conjunto de train, solo nos queda ver qué tal predice en el conjunto de test; para saber qué resultados podemos esperar a futuro de dicho modelo.

En clasificación multiclase las métricas como Precision, Recall, F1-Score, curva ROC... Son más difíciles de representar e interpretar, más que nada porque no tenemos Positives y Negatives como tal; sino 3 clases distintas. En los problemas de clasificación binaria (fuga/no-fuga, lloverá/no-lloverá) está claro que una cosa es lo que llamamos casos positivos y la otra los negativos; pero cuando tenemos tres...

9. Guardar el modelo/pipeline para futuros usos y hacer predicciones¶

Y poco más: una vez que hemos visto lo que podemos esperar del modelo a futuro, procedemos a guardar nuestro modelo/pipeline entrenado (por ejemplo, serializando el objeto y guardándolo en el disco duro).

Siempre que queramos hacer predicciones, no tendremos más que copiar el modelo en el ordenador donde queramos hacerlas, (que tenga instalado Python, Numpy, Scipy, Pandas y Scikit-Learn) cargar los datos sobre los que queremos predecir, cargar nuestro modelo entrenado y aplicar su .predict() sobre los datos :)

Para guardar la pipeline ganadora entrenada, vamos a utilizar la biblioteca de serialización de la biblioteca estándar de Python pickle.

¿Qué es la serialización? Muy sencillo: en programación, la serialización consiste en coger un objeto que está en la memoria RAM de nuestro ordenador (por objeto entendemos cualquier cosa que podamos asignar a una variable), y lo exportamos en formato binario para guardarlo en el disco duro (o enviarlo por la red). Del mismo modo, la deserialización es el proceso inverso: importar un objeto serializado para poder usarlo de nuevo en nuestro código.

Serializar/guardar un modelo/pipeline de Scikit-Learn entrenado es tan sencillo como exportarlo con pickle:

In [34]:
import pickle

# Para exportar, usamos pickle.dump:
with open("mejor_pipeline_iris.model", "wb") as archivo_salida:
    pickle.dump(mejor_pipeline, archivo_salida)

Ya está guardado, en un archivo llamado mejor_pipeline_iris.model, que podemos copiar en un pendrive o mandar por Internet a otro ordenador. Contiene la pipeline ganadora entera entrenada.

Vamos a re-cargarla, suponiendo que estuviéramos en otro ordenador. Se carga también con pickle:

In [35]:
# Para importar, usamos pickle.load:
